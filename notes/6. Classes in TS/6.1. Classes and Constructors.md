### JS OOP Basics Recalled

In JS, you might be used to doing:

```javascript
class Car {
    constructor(name, model, year) {
        this.name = name;
        this.model = model;
        this.year = year;
    }
}
```

This works great, but it is better to be more explicit:

```javascript
class Car {
    name;
    model;
    year;

    constructor(name, model, year) {
        this.name = name;
        this.model = model;
        this.year = year;
    }
}
```

> **Why explicitly declare fields first?**
> It makes the code more readable, the class structure easily identifiable, and is a better practice overall. So we do the 2nd way from now on.

### Moving to TS

In TS, it's the same as the 2nd method, but we add **Access Modifiers** before the explicit field declarations.

#### Access Modifiers

1. **`public`**: Can access (read & write) the property outside the class and inside subclasses. (Default if not specified)
2. **`private`**: Can only access the property inside that class itself. Subclasses and outside code cannot access it.
3. **`protected`**: Similar to `private`, but subclasses can also access it. Outside code still cannot.
4. **`readonly`**: Value can only be set during declaration or inside the constructor. Cannot be changed afterwards, even by class methods.

#### Example Class in TS

```typescript
class Human {
    public name: string;
    private gender: string; // Must annotate the type
    protected age: number;

    constructor (name: string, gender: string, age: number) { // Type parameters in constructor
        this.name = name;
        this.gender = gender;
        this.age = age;
    }

    changeAgeGender() { // public method by default
        this.age = 30;
        this.gender = 'female';
    }
}

// Instantiating the class
const Adam = new Human("Adam", "male", 22);

// Accessing properties
console.log(Adam.name); // Works! (public) -> "Adam"

Adam.gender; // Error: Property 'gender' is private
Adam.age;    // Error: Property 'age' is protected

Adam.changeAgeGender(); // Works! Internal method can mutate private/protected properties
```

#### Private vs. Protected

If a class `MetaHuman` extends `Human`, it inherits `public` and `protected` members, but **not** `private` ones.

```typescript
class MetaHuman extends Human {
    constructor(name: string, age: number) {
        super(name, "secret", age); // Must call super() with parent's required params
    }

    testAccess() {
        console.log(this.name); // Works (public)
        console.log(this.age);  // Works (protected)
        console.log(this.gender); // Error: Property 'gender' is private and only accessible within class 'Human'
    }
}
```

#### The `readonly` modifier

Placed after the access modifier. It makes the property immutable after initialization.

```typescript
class Vehicle {
    public readonly brand: string;
    
    constructor(brand: string) {
        this.brand = brand;
    }
}
```

> [!TIP]
> **Gotcha**: TS access modifiers (`public`, `private`, `protected`) only exist at compile-time for type safety. At runtime, they compile to normal JS properties. If you want true runtime privacy, use ES2020 private fields (e.g., `#gender`).
